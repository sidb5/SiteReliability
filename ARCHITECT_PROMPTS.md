# Architect Prompts — Intelligent Observability & Event Watchdog
## Vibe Coding Session Audit Log

This document captures the architectural direction and decisions made by the
Lead Architect throughout the design phase of the Watchdog system. It demonstrates
the thinking process, the questions raised, and the design challenges identified
before a single line of code was written.

## Phase 2 — Caching, Versioning & Maintainability

**Prompt : Caching Strategy**
> Did you think about Redis or other local caching? Also did you think about
> maintainability like versioning of the APIs?

**Prompt : Log Ingestion Architecture Challenge**
> This is a log analysis tool which means it is not
> generating lots of writes. The angle you need to think about is the quickness
> of reading the external logs that are being generated and detecting anomalies
> in a timely manner.
>
> Also think about how we are going to plug in to whatever logs we are monitoring.
> The app will need a setup page. Also because this is an alarms and anomalies
> generating service we need to support a webhook mechanism so that users of the
> service do not only have to poll by calling our APIs but can also subscribe and
> receive anomalies pushed to them.
>
> Also think about what anomalies we will detect and structure the JSON output
> contract accordingly.
>
> Also think about the optimal way to scan external logs — those logs can be in
> a database or as files, and if they get generated as files then after one file
> is completed apps generally generate new files. Do we want to support both
> scenarios or just one?
>
> How are you planning to read these external logs? The tables may be huge so
> what queries will you use? Are you going to do streaming or polling? If so how
> frequent? Should that be configurable in the setup panel?
>
> Also keep CLAUDE.md thin and move all architecture into a separate
> ARCHITECTURE.md file so we do not overwhelm Claude's context memory.

## Phase 3 — Multi-Tenancy & Data Architecture

**Prompt : Raw Log Storage Challenge**
> Have we listed the feature set somewhere? Because having a database means we
> are adding persistence functionality for the analysis and alarms our app is
> generating.
>
> Also I do not think we should be saving the log entries we read from external
> sources into our own database — it is wasteful and our database will get bloated
> very quickly. We only need to save anomalies, and perhaps for not more than 30
> days, and we should make that configurable. How do we delete anomaly entries
> that get older than 30 days?
>
> Also the logs generated by this app itself should be minimal and configurable
> by log level.

**Prompt : Multi-Tenancy Architecture Decision**
> Are we going to allow this service to only connect to one external platform
> for their logs or are we going to support connecting to multiple? Should our
> architecture be designed to support connecting to multiple and allow each user
> to configure their own connection? And should we essentially move the source
> configuration from admin level down to user level since it is a developer-facing
> concern?

## Summary of Architectural Challenges Raised

The following is a consolidated list of every architectural concern raised during
the design phase, demonstrating the depth of thinking applied before implementation
began.

**Functional Architecture**
- API-first compliance as a first-class design constraint, not an afterthought
- Plugin connector architecture to support heterogeneous log source types
  (file-based with rotation, database with high-water mark, push-based)
- Six-category anomaly taxonomy mapping to real SRE failure modes
- Versioned and stable anomaly output JSON contract for external consumers
- Webhook push delivery as a first-class feature alongside REST pull

**Data Architecture**
- Deliberate decision not to store raw log entries (write volume proportional
  to anomaly count, not log volume)
- Configurable anomaly retention with age-based cleanup excluding open alerts
- High-water mark polling pattern for efficient large-table reads
- Adaptive polling interval to respect source infrastructure

**Multi-Tenancy**
- Multi-tenant SaaS model designed from day one rather than retrofitted
- TenantContext dependency injection pattern to eliminate IDOR vulnerabilities
- Per-tenant source configuration, user management, and API key self-service
- Row Level Security evaluated, documented, deferred to PostgreSQL migration

**Security**
- Three-tier secret storage taxonomy: never stored / hashed / encrypted
- API key prefixed format (wdog_live_ / wdog_test_) for automated secret scanning
- Distinction between API key storage (SHA-256 hash) and webhook secret storage
  (Fernet encryption) — different because webhook secrets must be retrieved
- Zero-downtime API key rotation with configurable grace period
- Sensitive header redaction in all structured log output

**Caching & Performance**
- CacheBackend abstraction: in-process default, Redis-swappable via config
- EWMA state write-through cache to avoid DB write per ingest event
- Dashboard aggregation query cache to reduce DB load under auto-refresh

**Maintainability**
- URL path API versioning with router-level and schema-level namespace separation
- Deprecation header middleware stub for future version sunset signaling
- Jinja2 + Alpine.js + Chart.js frontend: Python-ecosystem templating, no second
  server, no build pipeline, documented Next.js upgrade path
- Configurable app log level adjustable at runtime without restart
- Module-by-module build order with explicit test requirements per module
- EWMA parameters tunable per source without redeployment

**Observability (Self-Monitoring)**
- Structured JSON logging with sensitive field exclusion
- Health endpoint reporting DB status, active sources, cache hit rate, log level
- Request audit log per tenant for abuse investigation
- Eval targets with numeric thresholds for anomaly detection quality
